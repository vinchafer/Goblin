## 10. EDIT 8 — Stop button + abort

**Part A (ChatInput.tsx):** new props `isStreaming?`, `onStop?`. Render:
```tsx
{isStreaming
  ? <button onClick={onStop} className stop> <rect filled square/> </button>
  : <button onClick={submit} disabled={!hasInput} ...gold/muted up-arrow... />}
```
Stop: 32×32, r8, bg `--rule-strong`, color `--ink-deep`, lucide Square (filled). `submit` now guarded by `isStreaming` so a second send can't fire mid-stream; textarea stays editable (no `disabled` during stream).

**Part B (standalone-chat.tsx):** `abortRef` already existed and is passed to `apiStream(..., abortRef.current.signal)`. Added:
```tsx
const handleStop = () => {
  abortRef.current?.abort();
  streamingMsgRef.current = null;
  setIsStreaming(false);
};
```
Catch now treats AbortError as non-error and keeps the partial:
```tsx
if (err instanceof Error && err.name === "AbortError") {
  streamingMsgRef.current = null;
  setIsStreaming(false);
  return;   // partial message stays in messages[]
}
```
ChatInput call: removed `disabled={isStreaming}`, added `isStreaming={isStreaming} onStop={handleStop}`.

## 15. Surprising

Two items. (1) **Retired-token grep is not zero repo-wide**, but every hit is a pre-existing static asset under `apps/web/public/` (PWA manifest, offline page, brand wordmark/social SVGs) still carrying the old `#2D4A2B` green and DM Sans/Fraunces. (2) **Streaming caret placement:** react-markdown v9 renders content as block `<p>`, so the trailing `.caret` sits just after the last paragraph rather than perfectly inline at the last character (the mockup shows it inline). Functionally correct (blinks during stream), visually a hair below the last line in multi-paragraph replies — flagged as a minor fidelity gap, not fixed (would require injecting the caret into the markdown AST). Also: `react-markdown` was already at 9.0.1 (spec said ~6.0.x); kept the installed version and used the v9 `code` API.
